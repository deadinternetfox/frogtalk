# Contributing to FrogTalk

> **Our development process, in plain English:** Idea вҶ’ AI slop вҶ’ Deslop (human review) вҶ’ Ship.
>
> FrogTalk is largely vibe-coded. The first draft of almost every feature was generated by an AI assistant, then a human reviewer takes a pass to tighten the security, fix the dumb parts, and make sure it doesn't break anything else. We call that second pass **"deslopping."** It's the only thing that turns AI slop into production code.
>
> This document explains how to plug into that pipeline вҖ” whether you're filing a bug, sending a vibe-coded PR, or reviewing one.

## Why we're honest about this

A lot of projects use AI assistance and don't say so. We do say so, because:

1. **Users deserve to know the threat model.** If a feature was vibe-coded and hasn't been deslopped yet, that's a different risk profile than something a senior engineer wrote and a security firm audited. Don't pretend otherwise.
2. **It invites the right kind of help.** People who want to send a PR or file a CVE need to know we'll actually read it and act on it, not bury it.
3. **It sets the bar for contributors.** Vibe-coded PRs are welcome, but they don't get merged until a human has deslopped them. That's the deal. No "AI wrote it, looks fine, ship it" вҖ” we have to actually read the diff.

There's no shame in vibe-coding. There **is** shame in not reviewing the slop before it ships.

```
   "what if weвҖҰ"  whatever drafts a   refactor + actually        deploy to both
                  PR or commit         test the thing            nodes
```

Stages:

- **Idea.** Filed as a GitHub issue (any of the templates), a DM in `#dev`, or a TODO comment in code. No barrier вҖ” half-formed ideas are fine.
- **AI slop.** Anyone (maintainer or contributor) can have an AI generate the first draft and open a PR with it. **Tag the PR with the `vibe-coded` label** so the reviewer knows what they're walking into.
- **Deslop.** A human deslopper reviews the diff against [the checklist below](#what-deslopped-actually-means-the-review-checklist), pushes fix-ups, and removes the `vibe-coded` label once it passes. Add `deslopped` instead.
- **Ship.** Merge to `main`, deploy to both nodes via the existing `scp + systemctl restart` flow, watch the logs for a few minutes.

A PR with the `vibe-coded` label is **never auto-merged**. A human must deslop it first.

## How to file an issue

Use one of the issue templates:

- **Bug report** вҖ” something is broken or behaves wrong.
- **Security vulnerability** вҖ” see the dedicated process below.
- **Feature idea** вҖ” half-formed is fine; we'd rather have the idea written down than lost.
- **Slop sighting** вҖ” you spotted code that looks AI-written and obviously wrong (dead variables, misleading comments, suspicious string handling, unused dependencies). Tag it as a slop sighting; we'll deslop it.

Good issues:

- Have a clear, specific title (`"DM media URL renders raw HTML"`, not `"DMs broken"`).
- Include repro steps when behavioural.
- Link the relevant file/line if you can (GitHub permalinks are great).
- Don't include PII, session tokens, or other users' messages.

## How to report a security vulnerability

You have two channels:

1. **Public web form (preferred for most issues):** <https://frogtalk.xyz/security> вҖ” anonymous submissions accepted, rate-limited.
2. **Email for sensitive disclosures:** `security@frogtalk.xyz`. Please include a PoC, your suggested severity, and how you'd like to be credited.

Scope, threat model, and "what counts" are documented at the `/security` page.

**We don't run a bounty (yet).** What we offer instead:

- Public credit in the security advisory and on the Hall of Fame.
- Fast turnaround вҖ” most fixes ship within a week, often the same day for critical issues.
- An honest acknowledgement of what was broken and why. No PR theatre.

Please don't run destructive exploits against the live production server. Spin up a local instance with `docker compose up` and test there.

## How to submit a PR

1. **Fork** the repo and **branch from `main`**. One PR = one concern.
2. **If it's vibe-coded, say so.** Add the `vibe-coded` label, or just write `[vibe-coded]` in the PR title. We'd rather know upfront than discover it during review.
3. **Run the sanity checks** before pushing вҖ” see [the checklist](#what-deslopped-actually-means-the-review-checklist). At a minimum:
   - `node --check node/static/js/<file>.js` for any JS you touched. Silent parse errors break every onclick on the page.
   - `python -m py_compile node/<file>.py` for any Python you touched. The FastAPI app imports the whole tree at startup.
4. **Write a clear PR description** with:
   - What problem this solves (link the issue if there is one).
   - How you solved it (one paragraph is fine).
   - For security fixes: a PoC, or a link to the bug-report ID you filed at `/security` so the reviewer can verify the patch closes the hole.
   - What you didn't do (known gaps, follow-ups).
5. **Be patient with the deslop pass.** A maintainer will push fix-up commits or comment with specific changes. This isn't a rejection вҖ” it's the process. Vibe-coded PRs almost always need at least one deslop pass.
6. **Credit yourself** in the same PR by adding your handle to `CONTRIBUTORS.md`. We won't add it for you; that way you choose how you're named.

We use **trunk-based development** вҖ” there's one long-lived branch (`main`/`master`), and feature branches are short-lived. No `develop` branch, no release branches, no merge-commit ceremony unless something actually needs reverting.

## What "deslopped" actually means (the review checklist)

A PR is **deslopped** when a human reviewer has gone through this list and either marked each item вң… or filed a follow-up issue for it.

### Correctness

- [ ] **The code does what the PR says it does.** Run it. Don't just read it.
- [ ] **No dead code.** Variables defined but unused, branches that can never execute, comments describing behaviour that no longer exists вҖ” all gone.
- [ ] **No phantom imports.** Every `import` is actually used. Every dependency in `requirements.txt` / `package.json` is justified.
- [ ] **No copy-pasted comments that lie.** AI loves to leave aspirational docstrings. Either the comment matches the code or one of them changes.

### Security

- [ ] **All user-controlled values that hit the DOM are escaped** with `esc()` вҖ” and if they're going into a JS-string-in-an-HTML-attribute, with `jsStr()` instead. See [`node/static/js/state.js`](node/static/js/state.js) and the notes in `/memories/` for why `esc()` alone is unsafe inside `onclick="foo('${...}')"`.
- [ ] **All SQL uses parameterised queries.** No f-strings into `cursor.execute`.
- [ ] **All admin / state-changing endpoints check auth AND CSRF.** See the `_require_auth` pattern in [`node/routers/server_admin.py`](node/routers/server_admin.py).
- [ ] **All new endpoints validate input length and type.** Trust nothing from the wire.
- [ ] **No new secrets in the repo.** `git diff` against `.gitignore`d patterns; check for `.env`, service-account JSON, hardcoded tokens.
- [ ] **No new dependencies without a quick supply-chain check.** Look at the package's downloads, maintainer, last update, and whether the import name matches the package name (typosquat check).

### Operational

- [ ] **Migrations are idempotent.** `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE` guarded by a column-exists check. The migration runs on every startup.
- [ ] **Rate limits exist** on every endpoint that touches the DB or sends a notification. Anonymous endpoints get stricter limits.
- [ ] **Errors are logged but don't leak internals to the client.** No raw tracebacks in JSON responses.
- [ ] **The change is small enough to revert.** If it touches > 1000 lines, split it.

### Honesty

- [ ] **The commit message explains *why*, not just *what*.** "Refactor X" tells me nothing; "Refactor X because the WS handler held the SQLite lock during a remote fetch and blocked typing indicators" tells me everything.
- [ ] **Known gaps are listed in the PR description**, not buried in TODO comments.

If you can't tick every box, that's fine вҖ” file the gaps as follow-up issues with the `deslop-needed` label.

## Becoming a deslopper

Anyone can review PRs. The deslop team is whoever shows up and does the work consistently.

Path to write access:

1. Review 5+ PRs with substantive comments (not just "LGTM").
2. Ship at least one of your own PRs through the pipeline.
3. Ping a maintainer in the repo discussions and ask. We'll add you.

Deslopper responsibilities:

- Review at least one PR a week if you can.
- Be kind. The contributor probably did this in their spare time, possibly at 3am, definitely with an AI's help. Help them learn the codebase; don't dunk on the slop.
- When you push fix-up commits, write a short comment explaining what you changed and why. The contributor (and the next deslopper) needs to learn from the pass.
- If you're not sure whether something is safe, say so. "I don't know" is a valid review state.

## Credit, attribution, and conduct

- **Credit goes in `CONTRIBUTORS.md`.** Add yourself in your PR.
- **Security researchers** are credited in the release notes and on the [Hall of Fame](https://frogtalk.xyz/security#hall-of-fame).
- **Be excellent to each other.** See [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md) (Contributor Covenant 2.1). Report issues to `security@frogtalk.xyz` (subject: `Code of Conduct`) or via [Discussions](https://github.com/deadinternetfox/frogtalk/discussions).
