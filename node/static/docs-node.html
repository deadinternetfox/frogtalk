Self-Host A Federation Node

Production setup with update feed, legitimacy checks, onion-first routing, and federation-safe update flow.

0) Fast setup wizard (recommended)

Use the built-in CLI helpers for first-time setup and safe updates on self-hosted nodes.

git clone https://github.com/deadinternetfox/frogtalk.git /opt/frogtalk
cd /opt/frogtalk

# Interactive installer (setup · federation · updates · systemd)
bash node/scripts/install.sh

# Or direct commands:
bash node/scripts/install.sh setup -y
bash node/scripts/install.sh federation -y
bash node/scripts/install.sh update-apply

The wizard configures .env, installs dependencies, wires the runtime symlinks (node/data, node/.env, node/secrets), and can pull the latest git release on an existing checkout. Defaults the install path to your current clone (not only /opt/frogtalk).

0.5) Apply git updates (operators)

After setup, use the same installer menu for safe fast-forward updates — previews commits, refreshes venv deps, restarts frogtalk.service, and verifies /api/ping.

bash node/scripts/install.sh update
bash node/scripts/install.sh update-apply -y

# From a dev checkout (use your repo root, not /opt/frogtalk):
bash node/scripts/install.sh --install-dir "$(pwd)" update-apply

# Status shows how many commits you are behind upstream
bash node/scripts/install.sh status

Re-run install.sh federation -y after major releases if peer pills or mesh nav look stale. Hot SCP deploy (deploy_nodes.sh) does not run DB migrations — prefer update-apply when database.py changes.

1) Deploy FrogTalk Service

Clone the official repo, create a virtualenv, install requirements, and copy the env template.

git clone https://github.com/deadinternetfox/frogtalk.git /opt/frogtalk
cd /opt/frogtalk
python3 -m venv venv
source venv/bin/activate
pip install -r node/requirements.txt
cp node/deploy/env.example .env
$EDITOR .env   # set ADMIN_PASSWORD, PORT, ALLOWED_ORIGINS, PUBLIC_URL

2) Configure Federation + Update Channel

# /opt/frogtalk/.env
FROGTALK_FEDERATION_ENABLED=1
FROGTALK_FEDERATION_TOKEN=CHANGE_THIS_SHARED_SECRET
FROGTALK_OFFICIAL_DIRECTORY_URL=https://frogtalk.xyz/api/network/servers

# Signed inbox (default ON since 2026-05 hardening).
# Reject events from origins with no pinned server_pubkey.
FROGTALK_FEDERATION_REQUIRE_SIGS=1
# Pin peer keys from GET /api/network/status (directory has no pubkeys).
# Optional: federated voice/video + channel voice mesh
# FROGTALK_FEDERATION_CALLS_ENABLED=1
# FROGTALK_TURN_URLS=stun:stun.l.google.com:19302,turn:turn.example:3478
# FROGTALK_TURN_USERNAME=…
# FROGTALK_TURN_CREDENTIAL=…
# Inbox/outbox poll (seconds) — defaults 8 idle, 2 busy
# FROGTALK_FEDERATION_INBOX_IDLE_SEC=8
# FROGTALK_FEDERATION_OUTBOX_IDLE_SEC=8

# Centralized release feed from main site
FROGTALK_UPDATE_FEED_URL=https://frogtalk.xyz/api/network/updates/latest
FROGTALK_UPDATE_CHECK_INTERVAL_SEC=300

# Auto-apply is OFF by default (CRIT-3). Opt in explicitly.
FROGTALK_AUTO_UPDATE_ENABLED=0

# Ed25519 release signer pubkeys (hex), semicolon separated. Update
# apply refuses to run if the manifest signature isn't from one of these.
FROGTALK_RELEASE_SIGNERS=

# Public URL clients and federation peers should use (clearnet nodes)
PUBLIC_URL=https://your-node.example
# Optional alias — same role as PUBLIC_URL when set
# FROGTALK_BASE_URL=https://your-node.example

Use the same federation token only for trusted nodes under your control. Generate a strong value via python -c "import secrets; print(secrets.token_urlsafe(32))".

2.5) Run Behind Tor / Onion

If you want the node to be reachable over Tor, point clients and federation peers at the onion app URL, not the marketing home page.

# /opt/frogtalk/.env
FROGTALK_TOR_ENABLED=1
FROGTALK_ONION_URL=http://yourhiddenserviceaddress.onion
# Optional: keep empty for onion-only nodes so the public IP never appears in directory output
FROGTALK_BASE_URL=

# Tor Browser and other clients should open:
# http://yourhiddenserviceaddress.onion/app

# Clearnet hub pushing to onion peers needs local Tor SOCKS:
FROGTALK_TOR_SOCKS_PROXY=socks5://127.0.0.1:9050
Use /app for client handoff links and network-tab switches. The Node Control and network UI now prefer onion app links so users land inside the chat client, not the marketing site.

Calls in Tor Browser: voice/video need WebRTC. Tor Browser, private tabs, and some privacy extensions block it — use a normal window or the FrogTalk mobile app.

2.6) Join the mesh + federated calls

bash node/scripts/node_federation_join.sh --install-dir /opt/frogtalk -y
# Clearnet hub:
#   --public-url https://your-node.example
# Tor mirror:
#   --onion-url http://….onion --public-url ""
# Re-sync after deploys (safe to re-run):
bash node/scripts/node_federation_join.sh --install-dir /opt/frogtalk -y

Syncs frogtalk.xyz/api/network/servers (retries; falls back to the official Main + Tor mirror pair if unreachable), pins peer keys from /api/network/status, fixes node/data symlinks, and links board nav pills. Clearnet hubs need FROGTALK_TOR_SOCKS_PROXY to fetch .onion boards.

Cross-node calls: set FROGTALK_FEDERATION_CALLS_ENABLED=1, install coturn, and expose TURN in .env. Clients use GET /api/network/ice-config (authenticated). Spec: docs/FEDERATED_CALLS.md in the repo.

3) Start Service

sudo cp node/deploy/frogtalk.service /etc/systemd/system/frogtalk.service
# Defaults: WorkingDirectory=/opt/frogtalk/node, EnvironmentFile=/opt/frogtalk/.env
# Edit only the User if you're not running as `deploy`.
sudo systemctl daemon-reload
sudo systemctl enable frogtalk
sudo systemctl restart frogtalk
sudo systemctl status frogtalk --no-pager

4) Verify Node Health + Hash

curl -s https://your-node.example/api/network/status
curl -s https://your-node.example/api/network/build/local
curl -s https://your-node.example/api/network/update/status

`build_hash` should match the approved release from main-site feed for federation stability.

For onion-only nodes, verify the same endpoints through your Tor endpoint and test the user-facing app at http://yourhiddenserviceaddress.onion/app.

5) Publish Releases From Main Site (signed)

On frogtalk.xyz, publish a signed manifest once per release so nodes can verify and pull updates safely.

curl -X POST https://frogtalk.xyz/api/network/updates/publish \
  -H "X-Session-Token: ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "version": "1.5.4",
    "package_url": "https://frogtalk.xyz/releases/frogtalk-1.5.4.tar.gz",
    "package_sha256": "PUT_REAL_SHA256",
    "build_hash": "PUT_REAL_WEB_BUILD_HASH",
    "signature": "ED25519_SIGNATURE_OVER_CANONICAL_MANIFEST",
    "signer": "ED25519_PUBKEY_HEX",
    "notes": "Federation-safe rollout"
  }'

Operators set FROGTALK_RELEASE_SIGNERS in .env to the Ed25519 pubkey hex (semicolon-separated for multiple). Update apply refuses to run if the signature isn't from one of these signers — even if auto-apply is enabled.

6) Use Network Settings In App

Open app Settings -> Network to:

  • Probe nodes and view latency
  • Verify legitimacy badges (`SAME HASH`, `LEGIT COPY`, mismatch alerts)
  • Switch to Tor-enabled nodes using onion /app links with automatic handoff when possible
  • Check/apply updates from the official feed

8) Project structure (post 2026-05 restructure)

The repo now splits cleanly into client/ (what end-users install) and node/ (the federated server you run):

frogtalk/
├── client/                       # everything end-users install
│   ├── desktop/                  # Electron source + builds
│   └── mobile/{android,ios}
└── node/                         # the federated server
    ├── main.py                   # FastAPI entrypoint
    ├── routers/                  # FastAPI route modules
    ├── static/                   # web client + marketing pages
    ├── board/                    # Frog Channel PHP imageboard (public /board/)
    ├── deploy/                   # systemd / nginx / env.example
    ├── scripts/                  # setup, update, deploy, build helpers
    ├── tests/                    # pytest suite
    ├── requirements.txt
    └── Dockerfile

On a running node, operator state (.env, data/, secrets/, venv/) stays at /opt/frogtalk/ and the runtime source at /opt/frogtalk/node/. The setup wizard wires symlinks so the service can run with WorkingDirectory=/opt/frogtalk/node without copying secrets into the source tree.

Build entrypoints: client/desktop/app/package.json (Electron), client/mobile/android/build.gradle.kts, client/mobile/ios/FrogTalk.xcodeproj, and node/Dockerfile for containerized deploys.

The nginx file in git (node/deploy/nginx.conf) is a reusable template. Set your own server_name, cert paths, and upstream port to match your service override.
