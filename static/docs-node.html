Self-Host A Federation Node

Production setup with update feed, legitimacy checks, onion-first routing, and federation-safe update flow.

0) Fast setup wizard (recommended)

Use the built-in CLI helpers for first-time setup and safe updates on self-hosted nodes.

git clone https://github.com/deadinternetfox/frogtalk.git /opt/frogtalk
cd /opt/frogtalk
bash scripts/node_setup_wizard.sh

# later, check for updates
bash scripts/node_update_check.sh

# apply update safely (fast-forward only)
bash scripts/node_update_check.sh --apply

The wizard configures `.env`, installs dependencies, and handles common edge cases without aborting the whole flow.

1) Deploy FrogTalk Service

Clone the official repo, create a virtualenv, install requirements, and copy the env template.

git clone https://github.com/deadinternetfox/frogtalk.git /opt/frogtalk
cd /opt/frogtalk
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp deploy/env.example .env
$EDITOR .env   # set ADMIN_PASSWORD, PORT, ALLOWED_ORIGINS, PUBLIC_URL

2) Configure Federation + Update Channel

# /opt/frogtalk/.env
FROGTALK_FEDERATION_ENABLED=1
FROGTALK_FEDERATION_TOKEN=CHANGE_THIS_SHARED_SECRET
FROGTALK_OFFICIAL_DIRECTORY_URL=https://frogtalk.xyz/api/network/servers

# Signed inbox (default ON since 2026-05 hardening).
# Reject events from origins with no pinned server_pubkey.
FROGTALK_FEDERATION_REQUIRE_SIGS=1

# Centralized release feed from main site
FROGTALK_UPDATE_FEED_URL=https://frogtalk.xyz/api/network/updates/latest
FROGTALK_UPDATE_CHECK_INTERVAL_SEC=300

# Auto-apply is OFF by default (CRIT-3). Opt in explicitly.
FROGTALK_AUTO_UPDATE_ENABLED=0

# Ed25519 release signer pubkeys (hex), semicolon separated. Update
# apply refuses to run if the manifest signature isn't from one of these.
FROGTALK_RELEASE_SIGNERS=

# Public URL clients and federation peers should use (clearnet nodes)
PUBLIC_URL=https://your-node.example
# Optional alias — same role as PUBLIC_URL when set
# FROGTALK_BASE_URL=https://your-node.example

Use the same federation token only for trusted nodes under your control. Generate a strong value via python -c "import secrets; print(secrets.token_urlsafe(32))".

2.5) Run Behind Tor / Onion

If you want the node to be reachable over Tor, point clients and federation peers at the onion app URL, not the marketing home page.

# /opt/frogtalk/.env
FROGTALK_TOR_ENABLED=1
FROGTALK_ONION_URL=http://yourhiddenserviceaddress.onion
# Optional: keep empty for onion-only nodes so the public IP never appears in directory output
FROGTALK_BASE_URL=

# Tor Browser and other clients should open:
# http://yourhiddenserviceaddress.onion/app
Use /app for client handoff links and network-tab switches. The Node Control and network UI now prefer onion app links so users land inside the chat client, not the marketing site.

3) Start Service

sudo cp deploy/frogtalk.service /etc/systemd/system/frogtalk.service
# edit WorkingDirectory and User in the unit file if needed
sudo systemctl daemon-reload
sudo systemctl enable frogtalk
sudo systemctl restart frogtalk
sudo systemctl status frogtalk --no-pager

4) Verify Node Health + Hash

curl -s https://your-node.example/api/network/status
curl -s https://your-node.example/api/network/build/local
curl -s https://your-node.example/api/network/update/status

`build_hash` should match the approved release from main-site feed for federation stability.

For onion-only nodes, verify the same endpoints through your Tor endpoint and test the user-facing app at http://yourhiddenserviceaddress.onion/app.

5) Publish Releases From Main Site (signed)

On frogtalk.xyz, publish a signed manifest once per release so nodes can verify and pull updates safely.

curl -X POST https://frogtalk.xyz/api/network/updates/publish \
  -H "X-Session-Token: ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "version": "1.5.4",
    "package_url": "https://frogtalk.xyz/releases/frogtalk-1.5.4.tar.gz",
    "package_sha256": "PUT_REAL_SHA256",
    "build_hash": "PUT_REAL_WEB_BUILD_HASH",
    "signature": "ED25519_SIGNATURE_OVER_CANONICAL_MANIFEST",
    "signer": "ED25519_PUBKEY_HEX",
    "notes": "Federation-safe rollout"
  }'

Operators set FROGTALK_RELEASE_SIGNERS in .env to the Ed25519 pubkey hex (semicolon-separated for multiple). Update apply refuses to run if the signature isn't from one of these signers — even if auto-apply is enabled.

6) Use Network Settings In App

Open app Settings -> Network to:

  • Probe nodes and view latency
  • Verify legitimacy badges (`SAME HASH`, `LEGIT COPY`, mismatch alerts)
  • Switch to Tor-enabled nodes using onion /app links with automatic handoff when possible
  • Check/apply updates from the official feed

7) Repo structure for operators

As of the 2026-05 restructure, client surfaces live under client/ so node operators have a cleaner picture of what they're shipping:

client/
├── desktop/app/        # Electron source (legacy: desktop/)
├── desktop/builds/     # Electron build outputs (gitignored)
└── mobile/
    ├── android/        # Android source (legacy: android/)
    └── ios/            # iOS source (legacy: ios/)

backend/                # API/runtime boundary docs
node/                   # node-operator boundary docs
deploy/                 # systemd/nginx/env templates
scripts/                # node_setup_wizard.sh, node_update_check.sh

Legacy top-level desktop/, android/, ios/ are now compatibility symlinks pointing into client/, so any existing scripts you have keep working.

7) Project structure (2026 layout)

Current repository structure separates operator, backend, and client surfaces:

frogtalk/
├── node/                    # operator docs/workflows
├── backend/                 # backend boundary docs
├── client/                  # desktop/mobile surfaces
│   ├── desktop/app
│   └── mobile/{android,ios}
├── static/                  # web app + site pages (runtime path)
└── scripts/                 # setup/update/deploy helpers

Legacy root paths (`desktop`, `android`, `ios`) are compatibility symlinks so older tooling keeps working while migration finishes.
